# 依赖收集

# 概述

依赖收集是vue中实现响应式数据的关键一步，依赖收集的目的是为了数据变化时派发更新。本文主要讲述一下几个问题

  • vue是如何实现依赖收集
  • 依赖收集中的两个关键类 Watcher和Dep
  • vue在依赖收集的过程中所作的优化。

# 依赖收集的实现

在前面的文章中我们介绍了响应式数据的实现，主要是通过给数据添加存储器属性getter和setter实现依赖收集和 派发更新。在本文中我们重点介绍依赖收集。

首先我们在来回顾Vue实例或者组件实例的初始化过程。 初始化在initState中将props和data中的属性通过defineReactive方法变成响应式，也就是data和props中的 属性具有getter和setter。然后进入组件的mountComponent方法。

在mountComponent方法中定义了updateComponent方法，updateComponent方法会作为 渲染watcher的参数在实例化过程中会执行。

export default class Watcher {
  vm: Component;
  expression: string;
  cb: Function;
  id: number;
  deep: boolean;
  user: boolean;
  lazy: boolean;
  sync: boolean;
  dirty: boolean;
  active: boolean;
  deps: Array<Dep>;
  newDeps: Array<Dep>;
  depIds: SimpleSet;
  newDepIds: SimpleSet;
  before: ?Function;
  getter: Function;
  value: any;

  constructor (
    vm: Component,
    expOrFn: string | Function,
    cb: Function,
    options?: ?Object,
    isRenderWatcher?: boolean
  ) {
    this.vm = vm
    if (isRenderWatcher) {
      vm._watcher = this
    }
    vm._watchers.push(this)
    // options
    if (options) {
      this.deep = !!options.deep
      this.user = !!options.user
      this.lazy = !!options.lazy
      this.sync = !!options.sync
      this.before = options.before
    } else {
      this.deep = this.user = this.lazy = this.sync = false
    }
    this.cb = cb
    this.id = ++uid // uid for batching
    this.active = true
    this.dirty = this.lazy // for lazy watchers
    this.deps = []
    this.newDeps = []
    this.depIds = new Set()
    this.newDepIds = new Set()
		this.getter = expOrFn
		this.value = this.lazy
      ? undefined
      : this.get()
  }

  /**
   * Evaluate the getter, and re-collect dependencies.
   */
  get () {
    pushTarget(this)
    let value
    const vm = this.vm
    try {
      value = this.getter.call(vm, vm)
    } catch (e) {
      if (this.user) {
        handleError(e, vm, `getter for watcher "${this.expression}"`)
      } else {
        throw e
      }
    } finally {
      // "touch" every property so they are all tracked as
      // dependencies for deep watching
      if (this.deep) {
        traverse(value)
      }
      popTarget()
      this.cleanupDeps()
    }
    return value
  }

  /**
   * Add a dependency to this directive.
   */
  addDep (dep: Dep) {
    const id = dep.id
    if (!this.newDepIds.has(id)) {
      this.newDepIds.add(id)
      this.newDeps.push(dep)
      if (!this.depIds.has(id)) {
        dep.addSub(this)
      }
    }
  }

  /**
   * Clean up for dependency collection.
   */
  cleanupDeps () {
    let i = this.deps.length
    while (i--) {
      const dep = this.deps[i]
      if (!this.newDepIds.has(dep.id)) {
        dep.removeSub(this)
      }
    }
    let tmp = this.depIds
    this.depIds = this.newDepIds
    this.newDepIds = tmp
    this.newDepIds.clear()
    tmp = this.deps
    this.deps = this.newDeps
    this.newDeps = tmp
    this.newDeps.length = 0
  }

  /**
   * Subscriber interface.
   * Will be called when a dependency changes.
   */
  update () {
    /* istanbul ignore else */
    if (this.lazy) {
      this.dirty = true
    } else if (this.sync) {
      this.run()
    } else {
      queueWatcher(this)
    }
  }
	/**
   * Depend on all deps collected by this watcher.
   */
  depend () {
    let i = this.deps.length
    while (i--) {
      this.deps[i].depend()
    }
  }

  /**
   * Remove self from all dependencies' subscriber list.
   */
  teardown () {
    if (this.active) {
      // remove self from vm's watcher list
      // this is a somewhat expensive operation so we skip it
      // if the vm is being destroyed.
      if (!this.vm._isBeingDestroyed) {
        remove(this.vm._watchers, this)
      }
      let i = this.deps.length
      while (i--) {
        this.deps[i].removeSub(this)
      }
      this.active = false
    }
  }
}

export function defineReactive (
  obj: Object,
  key: string,
  val: any,
  customSetter?: ?Function,
  shallow?: boolean
) {
  const dep = new Dep()

  const property = Object.getOwnPropertyDescriptor(obj, key)
  if (property && property.configurable === false) {
    return
  }

  // cater for pre-defined getter/setters
  const getter = property && property.get
  const setter = property && property.set
  if ((!getter || setter) && arguments.length === 2) {
    val = obj[key]
  }

  let childOb = !shallow && observe(val)
  Object.defineProperty(obj, key, {
    enumerable: true,
    configurable: true,
    get: function reactiveGetter () {
      const value = getter ? getter.call(obj) : val
      if (Dep.target) {
        dep.depend()
        if (childOb) {
          childOb.dep.depend()
          if (Array.isArray(value)) {
            dependArray(value)
          }
        }
      }
      return value
    },
    set: function reactiveSetter (newVal) {
      const value = getter ? getter.call(obj) : val
      /* eslint-disable no-self-compare */
      if (newVal === value || (newVal !== newVal && value !== value)) {
        return
      }
      /* eslint-enable no-self-compare */
      if (process.env.NODE_ENV !== 'production' && customSetter) {
        customSetter()
      }
      // #7981: for accessor properties without setter
      if (getter && !setter) return
      if (setter) {
        setter.call(obj, newVal)
      } else {
        val = newVal
      }
      childOb = !shallow && observe(newVal)
      dep.notify()
    }
  })
}

从前面的分析我们可以知道,根实例或者组件实例都会对应相应的渲染watcher,每个渲染watcher在执行get方法时会执行updateComponent方法，updateComponent 方法会执行vm._update方法，在根实例的_update方法中通过vm._render生成整个vnode树，在执行根_实例update方法时会进入到组件的updateComponent 方法，也就是说根实例对应的渲染watcher的get方法还未执行结束此时会进入到子组件对应的渲染Watcher的实例化，在子组件渲染watcher实例化时，通过 pushTarget方法将Dep.target设置为子组件所对应的渲染Watcher，此时targetStack中包括了父渲染watcher和子组件渲染Watcher,当子组件Watcher实例化完成也就是执行完 updateComponent方法时，会执行popTarget,将子组件对应的渲染watcher从targetStack中删除，并将父渲染watcher设置为Dep.target。 当根实例update完成，同样会执行popTarget，从这里我们可以看到根实例渲染watcher实例化完成后，虚拟dom已经渲染完成，targetStack数组内没有任何渲染watcher

想一想vue为什么会设计一个Dep.target？ Dep.target在目前来看是一个渲染watcher(不考虑用户定义的watcher和computed)。Dep.target在依赖收集中会使用，会被添加到每个响应式数据对应的 dep.subs中。 每个响应式数据可能会被不同的组件所订阅也就是不同的渲染watcher。 那么响应式数据的getter在执行时是如何知道订阅者也就是当前的watcher是根实例App对应的watcher还是Child组件对应的watcher或者说是Parent组件对应的Watcher呢？ 毕竟在根实例实例化watcher的过程中可能会生成一系列的Watcher。

为了解决这个问题，vue设计了Dep.target,将Dep.target设置为当前正在实例化的watcher，当前的watcher在实例化时，会执行render函数，会访问响应式对象，触发getter, 并将Dep.target添加到响应式对象对应的dep的subs数组中作为其订阅者，这一过程也是为渲染watcher添加依赖，也就是说当前的渲染wactch依赖哪些响应式数据， 当这些响应式数据发生变化时会触发updateComponent方法，进行更新。

当前正在实例化的Watcher实例化完成会退出targetStack，Dep.target会被重新设置为targetStack的其他watcher，通过 Dep.target = targetStack[targetStack.length - 1]来实现。

通过上面的文字分析我们可以知道每个响应式数据的dep中可能会被添加多个渲染watcher，而每个渲染watcher会有不同的依赖也就是dep。 保存在watcher的deps中。watcher和依赖dep是多对多的关系。

  • 每个dep有多个订阅者watcher
  • 每个watcher依赖不同的dep

分析到这我们基本讲完了依赖收集的整个过程。简单说，依赖收集就是给响应式数据添加订阅者watcher。给响应式数据添加订阅者watcher是通过 依赖dep来实现。而每个响应式数据都有对应的唯一的dep实例，这个唯一的dep实例会有不同的订阅者watcher。

# Watcher

watcher实例关键有以下几点

  • deps: 旧依赖
  • newDeps： 新依赖
  • depIds： 依赖id
  • newDepIds： 依赖id集合
  • update: 更新方法，在派发更新时调用
  • cleanupDeps： 依赖清除
  • addDep：添加依赖
  • get：执行updateComponent方法

# Dep

  • addSub 添加订阅
  • removeSub 删除订阅
  • depend 建立watcher和dep联系
  • notify 派发更新

# 优化

cleanupDeps用于删除依赖,初始化渲染时访问了响应式数据msg，此时当前的渲染watcher会被添加到msg对应的dep的subs中，其他数据变化导致更新， 此时会进入新一轮的依赖收集，由于更新时render函数没有访问msg，因此newDeps中没有msg依赖没有没收集，更新完成会执行cleanupDeps方法，将watcher从 msg所对应的dep依赖中删除watcher，因此当msg发生变化时，根实例或者组件并不会更新，这是vue中在依赖收集过程中所作的优化。 响应式数据的变化不一定会导致组件更新。

# 总结

  • 依赖收集是在执行render函数生成vnode时进行，访问响应式数据触发getter
  • Dep.target在执行render函数的过程中唯一，以确保依赖准确添加
  • 每次更新完成，都会进行相关依赖清除，以提高性能
